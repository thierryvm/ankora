## 1. Transport & headers

**Déclaré** (SECURITY.md) : HTTPS + HSTS preload, CSP per-request nonce +
`strict-dynamic`, COOP/CORP `same-origin`, `X-Frame-Options: DENY`,
`Permissions-Policy` verrouillé, `Referrer-Policy: strict-origin-when-cross-origin`.

**Observé** :

- `next.config.ts:37-64` pose les headers statiques (X-Frame-Options DENY,
  HSTS preload **conditionné à la prod**, Permissions-Policy exhaustif,
  Referrer-Policy correct).
- `src/proxy.ts:23-52` construit un nonce `nanoid()` par requête, l'injecte
  dans `request.headers` (pour les Server Components) et `response.headers`
  (pour le navigateur). `strict-dynamic` activé en prod uniquement (dev
  assoupli pour Turbopack HMR).

**Verdict** : A. CSP nonce + strict-dynamic correctement câblés. HSTS preload
bien conditionné à la prod (pas de lock-in local).

## 2. Auth & session

**Déclaré** : Supabase PKCE, mots de passe ≥ 12 caractères mixtes+chiffre,
rate limit 5 / 15 min par IP, MFA TOTP, session auto-refresh middleware.

**Observé** :

- PKCE activé dans `src/lib/supabase/server.ts:28` (`flowType: 'pkce'`).
- Rate limit Upstash dans `src/lib/security/rate-limit.ts:38-43`, fenêtre
  glissante **5 tentatives / 15 min**.
- `signupSchema` + resolver Zod imposent la politique de mots de passe.
- Session refresh dans `src/lib/supabase/middleware.ts:6-31` via `getUser()`
  à chaque requête.

**Point d'attention** : le rate-limit **dégrade silencieusement** si les
variables Upstash sont absentes (`console.warn` en dev, `console.error` en
prod, puis on laisse passer). Cf. P1 ci-dessous.

**MFA TOTP** : déclaré dispo en settings — UI pas encore auditée (phase
post-landing). À re-vérifier quand le feature sort.

**Verdict** : A (avec P1 actionnable sur Upstash).

## 8. Middleware surface

**Déclaré** : refresh auth, protection de route, rate-limit, injection headers.

**Observé** dans `src/proxy.ts` (export `proxy`, wiré comme middleware) :

1. i18n next-intl via `handleI18nRouting`.
2. CSP + nonce per-request (`buildCsp()` + `nanoid()`).
3. Refresh session Supabase (`updateSession()`).
4. **Pas de garde de route explicite** : la protection `/app/**` est
   gérée en Server Components via le layout auth.

**Gap** : c'est la best-practice Next.js App Router, mais cela impose une
**discipline sans faille sur le nesting des layouts**. Si une future route
`/app/nouvelle-feature/page.tsx` est créée en dehors du layout auth, elle
fuitera. Cf. P2 ci-dessous.

**Verdict** : B+. Le design est correct, le risque est opérationnel.

## Top risques priorisés

### P0 — aucun

Tous les baselines déclarés sont effectivement implémentés et vérifiés.

### P1 — Upstash Redis défaillant en prod = rate-limit silencieusement off

**Fichier** : `src/lib/security/rate-limit.ts:14-29`.

**Scénario** : si `UPSTASH_REDIS_REST_URL` ou `UPSTASH_REDIS_REST_TOKEN`
manque en prod (oubli de config Vercel, rotation ratée), le module log une
erreur console et **laisse toutes les requêtes passer**. Conséquence : brute
force auth non bridé, baseline "5 / 15 min" violée à l'insu de tous.

**Mitigation** :

- Court terme : ajouter un check au boot (`src/lib/env.ts`) qui **refuse de
  démarrer** en prod si les deux variables Upstash sont absentes.
- CI : step "verify-prod-env" qui échoue si les secrets Upstash ne sont pas
  définis dans l'environnement Vercel Production.
- Alerting : un endpoint `/api/health/rate-limit` qui retourne 503 si le
  client Upstash est null (monitorable via UptimeRobot gratuit).

**Effort** : ~45 min (1 PR ciblée).

### P2 — Protection de route implicite via layout

**Fichier** : `src/proxy.ts` + conventions `src/app/**`.

**Scénario** : un futur contributeur crée `src/app/app/nouvelle-page/page.tsx`
en dehors de l'arborescence sous layout auth. La page est rendue sans check.

**Mitigation** :

- Lint custom : ESLint rule maison qui vérifie que chaque `page.tsx` sous
  `src/app/app/**` a un ancêtre exportant un layout qui appelle
  `requireAuth()` (ou équivalent).
- Alternative : basculer une **partie** de la garde dans middleware (reject
  si `session` absent sur `/app/**`). Plus robuste mais complexifie le SSR.
- Test e2e qui crawle toutes les routes `/app/**` non authentifié et attend
  un redirect vers `/login`.

**Effort** : 1-2 h (le test e2e est le chemin le plus rapide).

### P2 — Observabilité logs : adopter un logger structuré

**Fichiers** : partout (console.\* épars).

**Scénario** : incident en prod. Les logs Vercel sont non structurés, pas de
`trace_id`, pas de niveau `warn/error/fatal`. Timeline d'incident
reconstruite à la main.

**Mitigation** :

- Adopter **Pino** (léger, gratuit, compatible Edge/Node). Un seul wrapper
  `src/lib/log.ts`, imports progressifs.
- Format JSON line, `trace_id` généré par le middleware, propagé dans les
  Server Actions via AsyncLocalStorage.
- Redaction automatique des clés PII (email, token, cookies) via
  `pino.redact`.

**Effort** : 3-4 h. Un ADR-004 serait justifié (choix durable,
dépendance tierce, changement transversal).

## Quick wins (< 1 h chacun)

1. **Pin Upstash en prod** : ajouter `UPSTASH_REDIS_REST_URL` et
   `UPSTASH_REDIS_REST_TOKEN` à la liste `required` de `src/lib/env.ts` en
   mode prod uniquement, via un bloc `if (process.env.NODE_ENV === 'production')`.
2. **Documenter la politique** : un paragraphe dans le README qui liste
   explicitement "5 auth attempts per 15 min per IP" et la conséquence en cas
   de dépassement. Transparence pour contributeurs.
3. **Security.md** : migrer `thierryvm@gmail.com` → `security@ankora.be` dès
   que le domaine le permet (déjà listé en TODO dans `SECURITY.md:13`).
4. **CSP report-uri** : ajouter une route `/api/csp-report` qui collecte les
   violations CSP (utile pour détecter un contournement ou un faux positif
   CDN). Stockage Supabase, RLS admin-only.

## Gaps stratégiques (nécessitent un ADR ou une PR dédiée)

1. **Logger structuré (ADR-004)** — cf. P2. Choix durable, impact
   transversal, dépendance tierce. Un ADR documente le choix (Pino vs
   Winston vs Upstash Log Collector) et le modèle de trace_id.
2. **Enforcement MFA** — SECURITY.md déclare MFA TOTP dispo mais pas
   enforced. À décider : MFA optionnel (UX permissive) vs MFA forcé pour
   comptes avec > 1 workspace ou > X opérations. Un ADR-005 peut trancher.
3. **Pentest régulier** — pas de calendrier défini. En mode projet vitrine,
   un pentest annuel (même en self-audit via OWASP ZAP scan automatisé
   en CI) est un marqueur de sérieux pour les candidats recruteurs.
